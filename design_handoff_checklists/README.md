# Handoff: Checklists — Sports Card Set Browser

## Overview
A page for browsing sports card sets ("checklists") across multiple sports. Users can filter by sport, search, toggle between a visual gallery and a compact list, see which sets were recently added, and drill into any set to see its card-by-card checklist with tracking checkboxes. Both desktop and mobile layouts are included.

## About the Design Files
The files in `/design/` are **design references created in HTML** — prototypes showing intended look and behavior, not production code to copy directly. The task is to **recreate these designs in the target codebase's existing environment** (React, Vue, SwiftUI, native, etc.) using its established patterns and libraries. If no environment exists yet, pick whichever framework fits the rest of the product.

The HTML prototypes use inline React + Babel for iteration speed. Production should use the codebase's component system, state management, styling solution, and icon library.

## ⚠️ Use Existing Imagery — Do Not Port the Placeholders
The card artwork you see in the prototypes (the colorful abstract shapes generated by `/design/card-art.jsx`) is a **placeholder only**. The production app already has real card-set imagery in its asset pipeline.

**Do this:**
- Use the app's existing card-set images wherever the prototype shows placeholder art — hero card on the detail screen, gallery tiles, compact-row thumbnails, and the mobile equivalents.
- Keep the prototype's **layout, sizing, aspect ratios (`2/2.8`), corner radii (`0` / square), shadows, hover transforms, and ribbon overlays** exactly as specified.
- The `CardArt` component in the prototype is a stand-in for an `<img>` (or the codebase's existing image component). Replace it 1:1; don't reimplement the procedural SVG.

**Do not do this:**
- Do not import `card-art.jsx` into production.
- Do not generate procedural art from set IDs.
- Do not invent new imagery — use whatever source (CDN, bundled assets, etc.) the rest of the app already uses.

The same applies to any sport/league logos, tier badges, or athlete photos if they appear — always prefer the existing asset source over anything in `/design/`.

## ⚠️ Use Existing Data — Do Not Port the Sample Content
Similarly, **all content in `/design/data.js` is placeholder data**. Set names (e.g. "2026 Apex Midnight UFC"), athlete/card counts, years, leagues, tiers, and featured flags were fabricated to populate the prototype. They must not ship.

**Do this:**
- Bind every field in the design to the corresponding field already in the production database / API:
  - Set name → existing checklist name
  - Sport category → existing sport field
  - Year → existing release year
  - Athlete count → existing athlete count
  - Card count → existing card count
  - Tier(s) / league(s) → existing tier and league fields
  - "Recently Added" ribbon → whatever existing field (or derived timestamp) the app already uses to flag recent additions; do not invent a new `featured` flag if an equivalent already exists.
- Keep the **design, layout, typography, and component structure** from the prototype exactly.

**Do not do this:**
- Do not use the fictional set names, counts, or other values from `data.js`.
- Do not create new schemas or fields to match the prototype's shape; adapt the design to the existing data model, not the other way around.
- Do not seed a database with this sample data.

Use `/design/data.js` only to understand what **types and shapes** of fields the UI needs, and map those onto the real schema.

## Interactions & Behavior

- **Filter**: selecting a sport chip filters the grid. Featured strip hidden when any filter or search active (desktop legacy — mobile has no strip).
- **Search**: case-insensitive substring match across set name, sport, league list, tier list, year.
- **View toggle**: `gallery` or `compact`; persisted to `localStorage` key `cl_view`.
- **Open set**: clicking a card/row sets `openId`. Full-page takeover replaces the index. Back button clears it. `openId` persists to `localStorage` key `cl_openId` so reload resumes.
- **Detail tabs**: local state only; does not change which cards render in the demo (production should load the appropriate card subset per tab).
- **Checkbox**: per-card collection tracking (mock — every 4th card pre-checked in demo).
- **Hover**: gallery cards translate up `2px` with elevated shadow; compact rows shift background.
- **Transitions**: `all 0.15s` on filter chips and compact rows; `all 0.2s` on gallery cards; `fadeInUp 0.3s` on detail page mount.

## State
- `view: 'gallery' | 'compact'` (persisted)
- `sport: string` (sport id, default `'all'`)
- `query: string`
- `openId: string | null` (persisted)
- `tweaks: { density: 4|5|6, showMeta: boolean }`
- `tab: string` (within detail)

Recommend: filters/search/view live in URL query string in production for shareability (e.g. `?sport=basketball&q=chrome&view=compact`).
